Kin-Achat : Spécifications Générales & Techniques

1. Vision & Structure Entreprise (Validé)
  • Institution Hybride : Présence physique (Gombe) pour la confiance, "cerveau digital" automatisé pour l'efficacité.
  • Diversification : Activités annexes (imprimerie/affichage) pour financer le local sans dépendre des revenus de l'application.
  • Positionnement : Facilitateur technologique, pas une banque. Sécurité par la résilience, pas par le marketing.
2. Logique Financière & Liquidité (Détails Pratiques)

Formule SDR (Solde Disponible Réel) : (Solde KeeCash + M_Encours) - Sum(Clients_Bloqués). Si ratio < 1.05, suspension automatique des nouveaux achats.

Gestion de la Latence : Inclusion des dépôts Mobile Money en attente dans le calcul SDR avec un coefficient de probabilité basé sur l'historique moyen (ex: 30 min).

3. Architecture Technique (n8n + Docker)

Workflows Clés :

  • Routeur d'Achat (W1) : Vérification solde + appel API KeeCash + Transaction SQL (Commit).
  • Gestionnaire Erreur (W2) : Centralisation de tous les "Error Trigger", logs table system_logs + alerte Telegram immédiate.
  • Réconciliation (W3) : Audit quotidien automatique.
4. Sécurité (Red Teaming & API)
  • Clés API : Utilisation de .env (non exposé dans le code). Séparation stricte des clés Lecture et Écriture.
  • Blindage : Authentification par token secret sur chaque Webhook d'entrée pour bloquer les appels non autorisés.
  • Red Teaming : Simulation hebdomadaire de pannes API et d'injections de données fausses.
5. Ce qui a été invalidé (Apprentissages)
  • Marketing de perfection : Invalidé au profit d'une transparence premium et d'une "Défense en profondeur".
  • Modèle bancaire traditionnel : Invalidé pour éviter les lourdeurs réglementaires et les risques de faillite centralisée.
  • Développement monolithique : Invalidé au profit d'une architecture modulaire par workflows spécialisés.
